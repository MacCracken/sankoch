## LOW (fixed in 2.0.0)

### LOW-01: `stream_compress_finish` / `stream_decompress_finish` didn't validate mode

**File**: `src/stream.cyr`
**Risk**: Calling the wrong finish on a mismatched-mode context would
load an inner pointer that was actually a buffer pointer (decompress
side) or vice-versa, dispatching to `deflate_enc_finish(buf_ptr)` and
crashing or emitting garbage.
**Fix**: Added mode check at the top of each finish:
`if (load64(ctx + 16) != STREAM_COMPRESS) return -ERR_INVALID_INPUT;`
Mirror check in `stream_decompress_finish`.
**Test coverage**: `test_stream_mode_mismatch` verifies both directions
return `-ERR_INVALID_INPUT`.

### LOW-02: dead accessors `_denc_load_level` and `_denc_state`

**File**: `src/deflate.cyr`
**Risk**: none — just dead code flagged by `cyrius build`. They were
placeholder helpers during the 1.7.0 encoder design; never called
from the final implementation.
**Fix**: Removed. `_denc_err` stays (used by `deflate_enc_write` and
`deflate_enc_finish` for the sticky-error check).

## INFO (documented, backlogged)

### INFO-01: `alloc()`-failure paths unchecked in `*_enc_init`

**Files**: `src/deflate.cyr` (`deflate_enc_init`), `src/zlib.cyr`
(`zlib_enc_init`), `src/gzip.cyr` (`gzip_enc_init`), `src/lz4.cyr`
(`lz4f_enc_init`), `src/stream.cyr` (`stream_compress_init`,
`stream_decompress_init`).
**Risk**: If `alloc()` returns 0 (heap exhausted / overflow),
subsequent `store64(ctx, ...)` writes to null and segfaults. The
mutex (taken in `deflate_enc_init` / `lz4f_enc_init`) would leak.
**Why not fixed now**: The pattern (unchecked `alloc`) is
project-wide, not new. Cyrius's bump allocator auto-grows the heap,
so failures are extremely rare. A consistent fix would touch many
call sites across the codebase.
**Backlog**: tracked as a v2.x hardening item ("defensive
alloc-failure handling"). Would require a sweep + helper pattern
(`_alloc_or_unlock_return(mutex_held, size)`).

### INFO-02: `adler32_update` is byte-at-a-time; lacks the 16-byte unroll from batch `adler32`

**File**: `src/checksum.cyr`
**Risk**: performance. Streaming zlib throughput is limited by the
byte loop in `adler32_update`. For 128 KB inputs in 4 KB chunks we
still do 128K individual `load8` + arithmetic ops on the checksum
side. The batch `adler32` uses a 16-byte unrolled inner loop.
**Why not fixed now**: `crc32_update` (used by gzip streaming) does
have the same unroll as its batch counterpart — the gap is only on
the zlib side. Fix is straightforward but not on the v2.0.0 critical
path. Backlogged as a v2.x perf item.

## Verified clean (highlights)

### DEFLATE streaming encoder (`deflate_enc_*`)

- **Window slide + LZ77 rebase** — 64 KB window, slides every 32 KB
  of new input, `lz77_rebase(delta)` subtracts from every hash entry
  and marks negatives `-1`. Verified by
  `test_deflate_enc_slide` (100 KB input, 4 KB chunks) and
  `fuzz_deflate_stream` (sizes up to 200 KB, random chunking, across
  all 4 levels). SIZE and byte-level content match batch output.
- **LOOKAHEAD reserve** — 258 bytes held back during `enc_write`;
  full window consumed on `enc_finish`. Match-finder always sees
  enough trailing bytes to find full-length matches.
- **BFINAL choreography** — `enc_write` emits sub-blocks with
  BFINAL=0 only; `enc_finish` emits a BFINAL=1 sub-block (dynamic)
  or a 5-byte BFINAL=1 stored-LEN=0 trailer (fixed). Empty input
  still produces a valid BFINAL=1 marker.
- **Sticky error + mutex discipline** — first error on `bw_write`
  or `_dyn_flush_subblock` stores in `ctx.err`; subsequent
  `enc_write` short-circuit-returns; `enc_finish` still releases
  `_sankoch_mtx` on error path. `test_enc_error_paths` exercises
  this: deliberately overflow a tiny dst, verify sticky error,
  confirm subsequent `deflate_compress` on the same thread works
  (lock was released).

### Incremental checksums

- **Adler-32**: deferred-modulo inner loop preserved; correct across
  NMAX-window boundaries (`test_adler32_incremental` byte-at-a-time
  0..65).
- **CRC-32**: same 8-byte unroll as batch (`test_crc32_incremental`).
- **xxHash32**: stripe-accumulator path for total_len ≥ 16;
  partial-stripe buffer (16-byte inline) carries across `update`
  calls; final short vs long path decided by cumulative length.
  `test_xxhash32_incremental` verifies byte-at-a-time updates 0..40
  (spans the stripe boundary) + multi-chunk combinations match the
  batch result. `xxh32sum` reference values pinned in
  `test_xxhash32_known_vectors`.

### MED-01 two-tier split

- Every batch public function that touches shared state takes
  `_sankoch_mtx` and delegates to an `_*_inner` variant:
  - `lz4_compress`, `lz4f_compress`
  - `deflate_compress`, `deflate_compress_level`,
    `deflate_decompress`, `deflate_decompress_dict`
  - `zlib_compress`, `zlib_compress_level`, `zlib_decompress`,
    `zlib_decompress_dict`
  - `gzip_compress`, `gzip_compress_level`, `gzip_decompress`
- `lib.cyr`'s `_compress_inner` / `_decompress_inner` call the
  `_*_inner` variants directly — avoids double-lock.
- `lz4_decompress` / `lz4f_decompress` intentionally lock-free —
  verified by reading; they touch no shared state.
- Streaming encoders take `_sankoch_mtx` at `enc_init` and release
  at `enc_finish`. Concurrent encoders serialize on `init`.

### Reference-CLI compatibility (still holds)

- `zlib_enc_*` output on 100 KB text decoded by Python's
  `zlib.decompress`, md5 matches.
- `gzip_enc_*` output decoded by `gunzip`, md5 matches.
- `lz4f_enc_*` output on 150 KB (multi-block) decoded by `lz4 -dc`,
  md5 matches.
- Batch SIZE lines byte-identical to 1.6.1 baseline (md5
  `085f17f1227b863a21597969dea9a74a` on the pre-streaming-lines
  subset).
