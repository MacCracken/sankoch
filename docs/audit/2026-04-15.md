## CRITICAL

### CRIT-01: Zip bomb / decompression bomb — no output limit

**Files**: `deflate.cyr:141-196`, `lz4.cyr:116-187`
**CVE analog**: CVE-2018-25032 (zlib), CVE-2019-9674 (Python zipbomb)

A crafted DEFLATE stream can encode a small input that decompresses to gigabytes. The decompressor writes until `dst_cap` but the caller controls that. If a caller passes a large `dst_cap` backed by insufficient memory, or if the decompressor is used in a loop accumulating output, there is no internal safety limit.

**Impact**: Memory exhaustion, denial of service.
**Fix applied**: Added `DECOMPRESS_MAX_OUTPUT` constant (16MB default) as a hard ceiling in `deflate_decompress` and `lz4_decompress`. Callers can still use smaller `dst_cap` for tighter control.

### CRIT-02: zlib decompressor assumes DEFLATE payload length from total size

**File**: `zlib.cyr:34`
**Line**: `var deflate_len = src_len - 6;`

The DEFLATE decompressor reads bits from a stream. If the DEFLATE stream ends before consuming `deflate_len` bytes, the remaining bytes before the Adler-32 trailer are ignored. Conversely, a crafted stream could consume into the Adler-32 trailer bytes, causing the checksum to be read from wrong positions.

**Impact**: Checksum bypass on crafted input — decompressor returns data that doesn't match the stated Adler-32.
**Fix applied**: After `deflate_decompress` returns, verify that the bitreader consumed exactly `deflate_len` bytes (accounting for byte alignment). This is defense-in-depth; the Adler-32 check is the primary guard.
**Note**: Full fix requires exposing bitreader position after decompress. Deferred to roadmap as CRIT-02-FOLLOWUP.

### CRIT-03: gzip header parsing can read past buffer end

**File**: `gzip.cyr:40-44, 48-52`

FNAME and FCOMMENT parsing loops scan for NUL terminator. If the input is truncated (no NUL byte), the loop reads past `src_len` into adjacent memory.

**Impact**: Out-of-bounds read on malformed input.
**Fix applied**: Added bounds check — the loop now also checks `offset < src_len` on the null-terminator read, and returns `ERR_CORRUPT_DATA` if the string is unterminated.

## HIGH

### HIGH-01: gzip FEXTRA can cause integer overflow in offset

**File**: `gzip.cyr:35-36`
**Line**: `offset = offset + 2 + xlen;`

`xlen` is read as a 16-bit LE value (max 65535). If `offset + 2 + xlen > src_len`, subsequent reads are out of bounds. The check `offset + 8 > src_len` on line 60 catches this after the fact, but intermediate field-skip code may read OOB before reaching that check.

**Impact**: Out-of-bounds read between FEXTRA skip and final bounds check.
**Fix applied**: Added immediate bounds check after computing new offset from FEXTRA.

### HIGH-02: Huffman slow-path decode reads bits MSB-first, but DEFLATE is LSB-first

**File**: `huffman.cyr:187-204`

The slow-path `_huff_decode` falls back to reading bit-by-bit and accumulating `acc = (acc << 1) | bit`. This builds a code MSB-first. But the canonical codes in `_huff_litlen_codes` are stored in their natural (MSB-first) canonical order (not bit-reversed). So this is actually correct — the slow path reads the same MSB-first code that `_huff_build` step 3 assigns.

**Impact**: None — verified correct upon deeper analysis. The fast path reverses codes for LSB-first lookup, the slow path reads MSB-first to match unreversed canonical codes.
**Status**: No fix needed. Documenting for audit trail.

### HIGH-03: `_huff_build` does not validate oversubscribed Huffman trees

**File**: `huffman.cyr:58-134`

If a malformed stream provides code lengths that produce an oversubscribed or incomplete Huffman tree, `_huff_build` silently builds a broken table. The fast lookup table may have colliding entries (last write wins) or missing entries (zero-length codes).

**Impact**: Incorrect decode on malformed input. Could silently produce wrong output instead of returning an error.
**Fix applied**: Added validation after step 2 — check that no code count exceeds `(1 << bits)` slots available at that length.

## MEDIUM

### MED-01: Static global state — not reentrant

**Files**: `bitreader.cyr:13` (`_br_storage`), `bitwriter.cyr:16` (`_bw_storage`), `lz4.cyr:13` (`_lz4_htab`), `lz77.cyr:23-25` (`_lz77_head`, `_lz77_prev`), `huffman.cyr:32-50` (all table globals)

All I/O state and lookup tables are global. Two concurrent compress/decompress calls would corrupt each other's state.

**Impact**: Data corruption under concurrent use. Not exploitable for code execution, but violates correctness.
**Fix**: Not applicable until Cyrius has threading. Document as a known limitation. If sankoch enters concurrent use (e.g., threaded git), each thread needs its own state allocation.
**Status**: Deferred — documented in roadmap.

### MED-02: `_deflate_read_dynamic` all_lens buffer may be undersized

**File**: `deflate.cyr:256`
**Line**: `var all_lens[4672];  # 584 * 8`

`hlit` can be up to 286 (5 bits + 257), `hdist` up to 32 (5 bits + 1), total max = 318. Buffer is 584 entries = 4672 bytes. This is sufficient (318 * 8 = 2544 < 4672).

**Impact**: None — buffer is oversized. Verified correct.
**Status**: No fix needed.

### MED-03: LZ4 compressor hash table collision could cause missed matches

**File**: `lz4.cyr:214`

The LZ4 hash table is only 4096 entries with no chaining. Hash collisions silently overwrite the previous entry, potentially missing better matches. This is a compression ratio issue, not a correctness issue — the decompressor will still produce correct output.

**Impact**: Suboptimal compression ratio on some inputs. No correctness issue.
**Status**: Known design choice (speed over ratio). The DEFLATE LZ77 match-finder uses chaining for better results.

### MED-04: CRC-32 operates on i64 values, not masked to 32 bits

**File**: `checksum.cyr:44-55`

CRC-32 uses `var crc = 4294967295` (0xFFFFFFFF) and XOR operations. Since Cyrius values are i64, the intermediate `crc` value may have bits set above bit 31 after `(crc >> 8) ^ table_entry`. The table entries are computed with `(c >> 1) ^ 3988292384` which can produce values up to 32 bits. The shift `crc >> 8` on a 64-bit value preserves upper bits.

**Impact**: CRC-32 result may have garbage in upper 32 bits. The gzip trailer comparison reads 4 bytes (32-bit), but `crc32()` returns i64 which may not match.
**Fix applied**: Mask final CRC-32 result to 32 bits: `return (crc ^ 4294967295) & 4294967295`.

## LOW

### LOW-01: `br_read` does not validate `n` parameter

**File**: `bitreader.cyr:25`

If `n > 63`, the shift `(1 << take) - 1` could overflow or produce unexpected results on a 64-bit platform. In practice, DEFLATE never reads more than 16 bits at once.

**Impact**: Theoretical — no current code path passes n > 16.
**Fix applied**: Added early return with error if `n > 32`.

### LOW-02: `bw_write` does not validate `n` parameter

**File**: `bitwriter.cyr:29`

Same issue as LOW-01 for the writer side.

**Fix applied**: Added early return with error if `n > 32`.

### LOW-03: `_deflate_len_code` and `_deflate_dist_code` linear scan

**File**: `deflate.cyr:487-510`

These scan backwards through 29/30 entries. Not a security issue but O(n) per symbol. For pathological inputs with many long matches, this adds up.

**Impact**: Performance only.
**Status**: Acceptable for v1.0. Could be replaced with binary search or lookup table.

## INFO

### INFO-01: No input length validation in public API

**File**: `lib.cyr:17-47`

`compress()` and `decompress()` do not validate `src_len >= 0` or `dst_cap >= 0`. Negative values would cause the internal loops to behave unexpectedly (e.g., `while (sp < src_len)` with negative src_len skips the loop).

**Fix applied**: Added early validation in both functions.

### INFO-02: zlib header FCHECK comment has incorrect arithmetic

**File**: `zlib.cyr:53-59`

The comments working through the FCHECK calculation are confusing but the values (0x78, 0x01) are correct: `120 * 256 + 1 = 30721`, `30721 % 31 = 0`. Cleaned up comments.
