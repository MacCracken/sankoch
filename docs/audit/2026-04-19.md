## HIGH

### HIGH-01: xxHash32 incomplete â€” short-length variant only, plus wrong prime in 4-byte tail

**File**: `src/checksum.cyr:150-200` (pre-fix)
**Impact**: LZ4 Frame content checksum diverged from reference xxHash32
for **any input â‰Ą 4 bytes** (and worse for â‰Ą 16 bytes). Our
compressor-produced frames would be rejected by the reference `lz4`
CLI with "Error : Stream checksum error", breaking the v1.6.0 promise
of "byte-identical output to `lz4` CLI on inputs >64KB".

**Two separate bugs in one function:**

1. **Missing 4-stripe accumulator path for `len â‰Ą 16`.** Reference
   xxHash32 initializes `v1=seed+P1+P2, v2=seed+P2, v3=seed, v4=seed-P1`
   and processes 16-byte stripes through `xxh32_round`, combining via
   `rotl(v1,1)+rotl(v2,7)+rotl(v3,12)+rotl(v4,18)` before adding `len`.
   Our code always used the short-length formula `h = seed + PRIME5`.

2. **4-byte tail used `PRIME2` instead of `PRIME4`.** The reference's
   4-byte tail step is `h = rotl(h, 17) * PRIME4`. Our code had
   `PRIME2` there, and a misleading comment claiming "PRIME4 step uses
   PRIME2 for short path" â€” that statement is false; reference uses
   PRIME4 in both short and long paths.

**Why self-tests didn't catch it:** our encoder and decoder both used
the same broken hash, so round-trip-through-sankoch tests passed.
Nothing in the suite cross-checked against reference xxHash32 output.

**Reproduction (pre-fix):**
```
  xxhash32("abcd", 4)       â†’  0x62a178ae  (reference: 0xa3643705)
  xxhash32(16Ă—0x00, 16)     â†’  0x04329a38  (reference: 0x8e022b3a)
```

**Fix applied in 1.6.1:**
- Added `XXH32_PRIME4 = 668265263` to the enum.
- Added `_xxh32_round(acc, input)` helper for the stripe step.
- Added `len >= 16` stripe path; short inputs still skip straight to
  the tail with `h = PRIME5`.
- Corrected the 4-byte tail multiplier to `PRIME4`.

**Test coverage added:**
- `test_xxhash32_known_vectors` â€” 9 reference vectors generated via
  `xxh32sum`, covering: empty, 1/3/4/7/15-byte short-path cases (the
  4-byte and 7-byte cases specifically pin the PRIME4 fix), exactly
  16 bytes (one stripe, no tail), 39 bytes ("Nobody inspectsâ€¦" â€”
  exercises 2 stripes + 4-byte tail + byte tail), 64 bytes (4 stripes,
  no tail).

**End-to-end validation:**
- Compressed a 150KB text input with `lz4f_compress`, produced a
  724-byte frame with 3 data blocks, and decoded byte-identically via
  reference `lz4 -dc`. MD5 of decoded output matches the original.
  Pre-fix, the same test fails with a checksum error from `lz4 -dc`.

**Wire-format break (semver note):** Frames written by sankoch
1.4.0â€“1.6.0 carry the divergent content checksum and will fail
verification under 1.6.1's decoder. Given that no downstream consumers
have shipped against the LZ4F path yet (all consumers listed in
CLAUDE.md are future/planned), the user approved this break. Flagged
loudly in the 1.6.1 CHANGELOG.

## MEDIUM

### MED-01: Direct-entry format APIs bypass the `_sankoch_mtx`

**Files**: `src/lib.cyr:43-112` (the mutex + `compress`/`decompress`),
plus every direct public entry:
  - `lz4f_compress`, `lz4f_decompress` (`src/lz4.cyr`)
  - `zlib_compress`, `zlib_compress_level`, `zlib_decompress`,
    `zlib_decompress_dict` (`src/zlib.cyr`)
  - `gzip_compress`, `gzip_compress_level`, `gzip_decompress`
    (`src/gzip.cyr`)
  - `deflate_compress`, `deflate_compress_level`, `deflate_decompress`,
    `deflate_decompress_dict` (`src/deflate.cyr`)
  - `stream_compress_init`/`_decompress_init`/`_write`/`_finish`
    (`src/stream.cyr`)

**Impact**: The public-API wrappers (`compress`/`decompress` in
`lib.cyr`) take `_sankoch_mtx` before calling the format-specific
functions. But every format-specific entry is *also* publicly exported
and has no lock of its own. Shared mutable state they touch:

- `_lz4_htab` (LZ4 hash table; cleared at the top of every
  `lz4_compress`)
- DEFLATE symbol buffers, bitwriter/bitreader contexts, Huffman
  work-area (`_hcl_ws`), encoder fixed-tree cache (`_huff_fixed_built`)
- Lazy-init flags `_deflate_tables_built`, `_huff_tables_allocd`,
  `_deflate_len_lookup_built`, `_deflate_dist_lookup_built` â€” each is
  a non-atomic check-then-set

**Race scenarios**:
- Two threads concurrently calling `lz4f_compress` corrupt `_lz4_htab`
  (one clears while the other is iterating).
- Two threads concurrently calling `zlib_decompress_dict` â†’
  `deflate_decompress_dict` can both observe `_deflate_tables_built == 0`
  and double-initialize the Huffman tables, racing on the heap
  allocations.

**Why not fixed in 1.6.1**: A correct fix requires a two-tier API per
format (public `X` takes lock â†’ inner `_X_inner` does the work),
because `zlib_compress_level` calls `deflate_compress_level` directly
(not through `compress_level`), and `gzip` does the same. A single
non-recursive mutex would deadlock if wrapped at every public entry.
This refactor aligns naturally with v1.7.0's `stream.cyr` rearchitecture
around a consume-N-emit-ready state machine â€” the streaming API will
need its own locking discipline anyway, and doing both at once is the
right shape.

**Today's mitigation**:
- Current tests and fuzz are single-threaded; no active exposure.
- CLAUDE.md lists all consumers (future git impl, ark, AGNOS kernel,
  shravan, tarang) as *planned* â€” no shipping multi-threaded user.
- **Recommendation added to roadmap**: the v1.7.0 track must land the
  public/internal API split as a precondition, before any consumer is
  told they can call `lz4f_compress` / `zlib_compress_level` / etc.
  directly under concurrency.

## LOW / cosmetic (all fixed in 1.6.1)

### LOW-01: `src/checksum.cyr` header claimed SIMD SSE2 acceleration

Line 5 previously read "Adler-32 uses SIMD acceleration (SSE2) for the
inner loop". The implementation is a 16-byte *scalar* unroll â€” Cyrius
5.4.7 has no inline-asm / intrinsic mechanism (per the roadmap's
deferred PCLMULQDQ CRC-32 note). Rewrote the header to describe what
the code actually does and noted that real SIMD awaits Cyrius asm.

### LOW-02: `src/types.cyr:35` missing trailing `;`

`DECOMPRESS_MAX_OUTPUT = 16777216` was the only enum entry in the
project without a terminating semicolon. Cyrius tolerated it; added
the `;` for consistency.

### LOW-03: `src/checksum.cyr` CRC-32 table references missed a space

10 instances of `crc32_table +i * 8` / `crc32_table +index * 8`
(missing space after `+`). Project style elsewhere uses the spaced
form. Fixed via `sed -i 's/crc32_table +/crc32_table + /g'`.

### LOW-04: `src/stream.cyr` usage header referenced nonexistent functions

The doc block showed `stream_compress_write(ctx, â€¦)` and
`stream_decompress_write(ctx, â€¦)`. The actual function is
`stream_write` (shared between modes). Updated the examples.

### LOW-05: `src/deflate.cyr:1140-1150` historical note was long

The adaptive-block-splitting comment explained the v1.4.0â†’v1.5.0
evolution across 10 lines. Condensed to 6 lines describing the current
behavior, without the version archaeology (the CHANGELOG carries that
history).

## INFO / future work (tracked, not blocking)

### INFO-01: `_deflate_build_len_lookup` / `_dist_lookup` use backward linear search

`src/deflate.cyr` (lines reported by internal review). One-time init,
~7.4K operations total. A forward iteration would be simpler and
trivially faster but this is not a hot-path issue. Backlogged; worth
fixing opportunistically when touching that section.

### INFO-02: Huffman tree depth clamped post-hoc, not built length-limited

`src/huffman.cyr:482` clamps per-symbol depths to `HUFF_MAX_BITS=15`
after DFS extraction from the two-queue tree. On pathologically skewed
frequency distributions the clamped lengths could violate the Kraft
inequality, which `_huff_build` would detect and reject with
`ERR_INVALID_HUFFMAN`. Not a correctness bug (errors surface cleanly)
and not exercised by 1360 fuzz iterations. A proper fix would adopt
the package-merge algorithm to build length-limited trees directly.
Backlogged as a v2.0.0 candidate.
